## Honest Feedback

First friction point: the task tool payloads are brittle, and one malformed/deceptively accepted call created a confusing aborted session and boulder reminder noise. I need to slow down around mutually exclusive `category` and `subagent_type` fields, especially when I am trying to satisfy many wrapper instructions at once. Second friction point: the plan contained a final-wave instruction to wait for explicit user okay, while the orchestrator rules required auto-continuation and completion. That contradiction should be surfaced earlier or normalized in the plan template, because it creates ambiguity at the exact finish line. Third friction point: evidence packaging can accidentally become over-broad when using glob-like mental models. The Task 6 summary initially included stale prior-plan evidence, and only a direct read caught it. The lesson is that verification evidence must be scoped semantically to the active plan, not just named similarly.

## Lessons Learned

- Contract-first diagnostics work benefits from a dedicated notepad decision file before source edits begin.
- `contextSize` needs explicit wording because its name can be misread as rendered context length.
- Additive summary classification is safer than replacing compatibility fields.
- Plugin-injected diagnostic metadata must be documented separately from public MCP input schema.
- Evidence indexes should be curated, not globbed from every historical `task-*` artifact in `.sisyphus/evidence`.
