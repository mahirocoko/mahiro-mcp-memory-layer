Three friction points stand out. First, I initially recommended next steps from partly stale context. The user had to call that out, and the correction improved the whole session. I should have treated the earlier local commits and docs as evidence to re-check before giving prioritization. It was still acceptable because the user kept saying to continue, but the operational risk increased. Third, the diagnostics work revealed that some fields existed because they sounded useful, not because tests proved their diagnostic meaning. `contextSize` was the clearest example. The good resolution was to make it meaningful instead of adding more explanation around a dead placeholder. Future sessions should keep that pattern: source truth first, field semantics second, docs last.

## Lessons Learned

- Re-audit source and package metadata before prioritizing when a human says the context feels stale.
- Diagnostic fields must carry real signal; otherwise docs can accidentally turn placeholders into false contracts.
- Scoped empty retrieval needs to preserve the attempted scope so users can distinguish scoped misses from global trace absence.
- Review freshness and workflow recency are different clocks; `updatedAt` should not become evidence freshness.
- Small memory-only hardening changes compound well when each one has focused tests and no boundary expansion.
