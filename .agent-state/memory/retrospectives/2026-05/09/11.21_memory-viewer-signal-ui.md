## AI Diary

I felt the session improve when I stopped treating the viewer as a generic filter form and started treating it as a memory judgement surface. The user's repeated pressure helped: first, they pointed out that seeing rejected records first was conceptually wrong; later, they questioned whether I was being too conservative visually; finally, they challenged my no-JS assumption by asking about debounce and throttle. Each challenge exposed a real design boundary. I also noticed my own tendency to over-protect the existing implementation. That was useful for safety, but not enough for a UI that needed to feel simpler and more human. Gemini was valuable, especially when it forced the exact distinction between no-JS GET forms, submit buttons, history pollution, and what a tiny script would actually cost. The review agents also mattered because they caught two problems I did not catch early: list-mode filter-before-limit and trace writes from viewer search. This session had a +100% weekly session change in derived pulse data, but the more important momentum was qualitative: the viewer now has a clearer mental model, safer data behavior, and a cleaner UI.

## Honest Feedback

Three friction points stood out. First, I initially underplayed the final Gemini review step after the security/list fixes. The user had to remind me that the final diff had not gone back through Gemini. That was a process gap: when Gemini is part of the decision loop, the final post-fix preview must be explicit, not optional. Second, I was too conservative on UI at first. The first polish pass was safe and technically correct, but the user correctly sensed that I was avoiding larger visual changes. The second pass improved because I accepted that “same UX, bolder UI” was still within scope. Third, the filter simplification discussion showed I should separate “no JS by current constraint” from “JS is impossible.” The right answer was not a blanket no; it was an architecture tradeoff about history behavior, progressive enhancement, and fallback.

## Lessons Learned

- For inspection tools, default information scent matters more than raw completeness. `Verified` as default is product behavior, not cosmetic preference.
- Read-only means no writes anywhere, including indirect observability writes like retrieval traces from viewer search.
- Local list UIs must filter before applying display limits when the filter is the user's primary intent.
- In server-rendered GET forms, removing every submit button is not equivalent to simplification; it can remove deterministic interaction and fallback.
- If using Gemini as a design/review partner, final post-fix preview should be part of the acceptance criteria.

## Next Steps

- Consider a small live screenshot/manual browser pass if the user wants pixel-level review beyond HTML/CSS and tests.
- Commit only if the user explicitly asks.
- If future search auto-submit is desired, design it deliberately as progressive enhancement with history semantics, not simple debounced `form.submit()`.
- Keep memory-viewer scoped to read-only local introspection; do not add review/mutation controls without a separate explicit plan.
