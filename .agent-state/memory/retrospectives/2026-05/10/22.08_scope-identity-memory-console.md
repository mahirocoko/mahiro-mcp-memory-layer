## AI Diary

I felt the shape of this session shift from a small UI question into a real product-boundary cleanup. At first, the work looked like a console affordance issue: why can't a verified memory be rejected, and can project names show in row metadata? But the user's instinct about strange project/container values exposed a deeper seam. I had to slow down and resist the easy move of rewriting every ID immediately. The important part was separating the durable-key risk from the user-visible mess. That led to a safer sequence: central policy first, read compatibility second, guarded rewrite third, actual local rewrite only after the user clarified that current data was disposable. I also noticed a verification trap during manual QA: the first console on port 4318 was stale and showed the retired memory viewer. Catching that mattered. If I had trusted the first browser snapshot, I would have falsely reported a broken UI or verified the wrong artifact. The session had good momentum, with the pulse showing a 7-day streak and a +100% weekly trend, but the useful lesson was not speed. It was keeping the memory product honest: canonical storage, compatibility at read boundaries, and a maintenance path that can be run without pretending old data never existed.

## Honest Feedback

Three friction points stood out. First, the repo had accumulated multiple overlapping concepts of container identity: worktree, directory, raw path, default, and global-like placeholders. The code handled enough of them to work, but the absence of a named policy made the console expose the inconsistency directly. Second, the console documentation lagged behind behavior. `MCP_USAGE.md` still called Browse read-only even after verified rejection became intentional, which would have confused future agents or users. Third, manual QA was almost derailed by a stale local server. That was a good reminder that browser verification needs to prove it is exercising the current source, not just any process listening on the expected port. The work improved the system substantially, but it also revealed that this package needs continued discipline around “memory-only” boundaries: every new runtime or console feature should explain whether it affects canonical records, derived indexes, traces, or only the UI projection.

## Lessons Learned

- Treat `projectId` and `containerId` as durable keys first, display labels second. Normalize generation paths before touching stored records.
- Compatibility reads are safer than immediate migrations when existing memory may matter.
- Maintenance commands that rewrite canonical logs should default to dry-run, report counts/changes, and reindex derived storage only under explicit apply.
- Manual browser QA should verify the served app identity; stale local processes can invalidate the test.
- Docs must move with behavior when a memory surface changes from read-only to read-mostly.

## Next Steps
