## AI Diary
I felt this session shift from a straightforward bugfix into a reliability hardening pass with multiple hidden seams. At first the issue looked like “Gemini is flaky,” but the code inspection made it obvious that part of the problem lived in our own abstractions: retries were missing on the direct async path, runtime selection was not consistently pinned, and polling/trace visibility made it harder to tell what had been configured versus what actually happened. I’m glad I did not stop at the first patch, because every Oracle-style review uncovered one more surface where the contract was still slightly dishonest. The docs mismatch around flattened async polling responses was a good reminder that working code can still be misleading software if the public reference lies. The Gemini CLI control work also reinforced a different lesson: convenience escapes like raw extra args are tempting, but the right move in a tool repo like this is nearly always a typed surface plus explicit constraints. I also noticed how much better the reasoning got once the prompt stopped being “fix Gemini” and became a split model of upstream runtime behavior versus local orchestration responsibilities. That framing made almost every later decision easier.

## Honest Feedback
Three friction points stood out. First, the repo has a lot of closely related orchestration surfaces, and it is easy to think a contract is changed everywhere when it has only been changed in one or two paths. That happened repeatedly with async polling, runtime pinning, and the new Gemini input fields. Second, the public contract lives partly in code and partly in README examples, which means “passing tests” is not enough if the docs lag behind. The Oracle reviews were useful mainly because they kept catching documentation or interface mismatches after the code already looked fine. Third, the upstream Gemini behavior is noisy enough that it constantly tries to pull attention toward the provider/CLI runtime, while the repo still had real local gaps that needed fixing. It took discipline to separate “we cannot fix upstream capacity or unauthorized tool behavior” from “we absolutely can improve our retry, runtime, and observability posture.” If I repeated this session, I would front-load the public contract review earlier so fewer doc fixes were discovered late.

## Lessons Learned
- Treat direct async worker tools as their own public API surface, not just thin wrappers.
- If shell serialization cannot round-trip an accepted value, the schema is too loose.
- Shell pinning is part of correctness, not just an implementation detail, when env-driven runtime switching exists.
- Retry telemetry is only truly useful when configured policy and terminal outcome are both visible.
- Gemini authoring quality improves dramatically when prompts include real file structure and explicit constraints.

## Next Steps
- Consider adding a higher-level `cliProfile` or similar convenience mode for authoring-safe Gemini jobs.
- Investigate watchdog/reconciler behavior for genuinely stale async `running` records.
- Decide whether direct async Gemini tools should expose a more opinionated authoring-safe preset by default or stay fully explicit.
