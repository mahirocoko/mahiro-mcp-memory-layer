Instead, I kept pushing until the behavior was proven in a live OpenCode run. That mattered. Once I saw the generic message events arrive without a meaningful session-start wake-up, the problem stopped being abstract. It became a concrete contract mismatch between the ideal lifecycle and the actual live path. That was satisfying, because the fix became smaller and more principled: do not rewrite the whole plugin, just tolerate the real event order. ## Honest Feedback
First, I lost some time to the OpenCode logging surface because `this._client` failures made the usual lifecycle logs unreliable. That forced me to add a stderr fallback, which was the right move, but it also exposed how brittle my initial debugging assumptions were. Second, the distinction between “the event exists in official contract docs” and “the event is visible in the runtime instance I am measuring right now” stayed slippery longer than I wanted. I had to keep restating that difference to avoid fooling myself with partial evidence. Third, the session sprawled across investigation, live probing, comparison against external architecture, implementation, cleanup, docs, and commit. I held it together, but it was easy for the thread to fragment. In the future I should create a crisper local checkpoint earlier once the first live mismatch is reproduced, because that gives the rest of the debugging session a tighter center of gravity.

## Lessons Learned
- A passing harness for `session.created` is not enough when the host runtime can legally or practically surface generic session-scoped message events first.
- Live `opencode run` behavior should be treated as a first-class verification path for the OpenCode plugin, not as a late confidence check.
- Debug instrumentation is most valuable when it is narrowly scoped, env-gated, and removable without touching the behavior fix.

## Next Steps
- Keep the generic-event wake-up fallback as the stable behavior unless upstream OpenCode guarantees earlier dedicated session-start delivery in all live paths.
- If future regressions appear, start with `MAHIRO_OPENCODE_PLUGIN_DEBUG_STDERR=1` before adding new ad hoc logs.
- Consider adding a higher-level package/integration test that models the generic-event-first live behavior even more explicitly.
