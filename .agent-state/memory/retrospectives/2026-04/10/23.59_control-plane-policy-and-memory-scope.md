## Honest Feedback
Three friction points stood out clearly. First, the repo posture around orchestration versus direct execution was conceptually strong but operationally ambiguous until I wrote it down in `AGENTS.md`. The system already had the right instincts, but without explicit wording it was too easy to confuse “delegate-first” with “must always use a worker,” which creates hesitation and wasted thought. Second, the memory scope issue exposed an uncomfortable truth: the memory layer looked ready from the outside, but a missing `userId` silently degraded the most important flows. That kind of partial readiness is more dangerous than an obvious failure because it encourages trust before the plumbing deserves it. Third, some of the documentation was lagging the product surface by just enough to create small but recurring doubts—especially around sync versus async worker calls and how MCP polling is supposed to feel in practice. None of these were catastrophic, but they all imposed cognitive tax. The positive side is that each friction point now has a concrete fix or at least a documented direction, which should lower the next session’s startup cost materially.

## Lessons Learned
- If a repo wants a control-plane/execution-plane architecture, the doctrine needs to say so explicitly; otherwise the behavior will stay inconsistent even when the code is good.
- Async UX improvements are not complete until the polling contract is self-describing in both code and docs.
- Memory systems should fail loud or bind scope completely; a “ready but incomplete” state can mislead maintainers.
- When a local default identity is needed, make it explicit and test-backed rather than implied through hidden assumptions.
