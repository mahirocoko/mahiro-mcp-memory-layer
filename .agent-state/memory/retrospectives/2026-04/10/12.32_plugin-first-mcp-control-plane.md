- Added plugin config-file loading with user, project, and environment precedence, then split the large runtime shell into backend, state, compaction, logging, and adapter modules.
- Fixed the plugin tool contract mismatch by serializing tool outputs to strings so OpenCode could consume `memory_context` and the shared memory tools without runtime crashes.

- Added automatic `AGENTS.md` instruction injection so the plugin no longer depends on a manual `instructions` entry in `opencode.json`.
- Forced `orchestrate_workflow` on the MCP path to normalize jobs onto `workerRuntime: "mcp"` and added persisted `workerRuntimes` metadata to traces/results so the AI-facing control plane is auditable.

## Key Code Changes
- Plugin config now supports user/project JSONC plus env override precedence.
- Plugin tool output now follows the OpenCode string-return contract.
- Source-checkout plugin path injects local MCP config through the plugin `config` hook.
- Plugin path injects packaged `AGENTS.md` automatically into `config.instructions`.
- MCP orchestration tool now forces normalized jobs onto `workerRuntime: "mcp"` and records `workerRuntimes` in trace/result metadata.

## Architecture Decisions
- Treat plugin as the main product surface; keep standalone MCP as fallback or plugin-injected capability rather than the default user story.
- Use official plugin package types where they fit, but retain local extension types where the upstream hook/tool contract is still narrower than real runtime behavior.
- Keep shell runtimes behind MCP worker tools for now, but stop letting the AI-facing orchestration path request shell directly.
- Prefer additive observability metadata over breaking persistence schema redesigns.

## What Could Improve
- The retrospective pulse-generation shell shape was too fragile on the first pass and cost time with a heredoc parsing error.
- Some helper logic like `asRecord` / `toNonEmptyString` is still duplicated across plugin runtime modules.
- The current orchestration UX still relies on the caller using async mode wisely to avoid timeout pain.

## Blockers & Resolutions
- **Blocked by official plugin contract mismatch** → resolved by using official types conservatively and adding a local extension layer.
- **Blocked by OpenCode tool runtime expecting strings** → resolved by serializing plugin tool results at the adapter boundary.
- **Blocked by missing orchestration surface in plugin-only path** → resolved by injecting local MCP config through the plugin `config` hook.
- **Blocked by ambiguous AI-facing shell fallback** → resolved by forcing `workerRuntime: "mcp"` at the MCP orchestration boundary and tracing it explicitly.

## Honest Feedback
Three friction points stood out. First, there was too much hidden ambiguity in what “plugin-first” meant until I forced every claim through real OpenCode runs. It is easy to say a plugin works because config resolves or because a unit test passes, but this session showed that the runtime truth is what matters. Second, the repo still makes some transport boundaries harder to read than they should be. The distinction between plugin-native behavior, plugin-injected MCP behavior, and shell-behind-MCP worker execution is now better than before, but I still had to hold too much of that model in my head while editing. That suggests the code and docs can become even clearer. Third, the timeout story is still more fragile than I’d like. I found a practical equilibrium—MCP-only control plane with async-first orchestration—but the ergonomics are not fully idiot-proof yet. If a caller insists on waiting in the wrong place, the system can still look more broken than it really is. That is a product and observability problem as much as a runtime problem, and it deserves a deliberate next pass rather than another incidental fix.

## Lessons Learned
- A plugin becomes trustworthy only when its runtime behavior is verified after restart, not when its types or tests look plausible.
- The official OpenCode `config` hook is the key seam for bringing instructions and MCP capability into a plugin-first workflow.
- “MCP-first control plane” and “shell-free backend” are different goals; separating them made planning much cleaner.
- Additive observability metadata is a low-risk way to improve debuggability without destabilizing old stored records.

## Next Steps
- Consider a small UX guardrail around sync orchestration usage so timeout-prone paths are harder to trigger accidentally.
- Evaluate whether duplicated plugin-runtime helpers should be centralized now that the module split is stable.
- Decide whether local source-checkout MCP injection should stay source-only forever or evolve into a package-aware path later.
