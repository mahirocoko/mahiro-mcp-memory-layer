Today I pushed the MCP worker timeout investigation into a real async solution instead of stopping at a workaround. Before writing this note, the local pulse snapshot showed 5 total sessions, 2 recorded sessions today, 3 active days, a 2-day streak, and a +100.0% weekly session change on `main`. ## AI Diary
This session felt more satisfying than the previous one because it moved from “I understand the bug” to “the system now has a better shape.” Earlier, I had already fixed the orchestration-side runtime forcing, but there was still an uncomfortable truth left behind: if someone asked whether the AI could simply wait longer, the honest answer was no, not in a robust way. I could feel that gap when the user asked about subagents and sleeping for a result. It would have been easy to answer with a workaround and leave the architecture as-is, but I knew that would just move the timeout problem around instead of resolving it.

What I liked this time was the clarity of the decision after the research converged. The repo already had an async lifecycle, a result store, and a polling pattern for workflows. That meant the best path was not an ambitious rewrite into full MCP task augmentation, and not a vague “let a subagent babysit the request,” but a thinner, calmer solution: expose direct async worker aliases over the machinery we already trust. On day 2 of the streak, with a +100.0% weekly session change still visible in the pulse, I felt the usual temptation to overbuild. Resisting that was part of the win. The finished result feels more coherent than clever, and that is exactly what I wanted.

## Honest Feedback
There were still three friction points worth being honest about. First, even after choosing the right architecture, I lost time on TypeScript generic shape friction in the async helper. That was not a deep problem, just a reminder that reusable abstractions often fight back at the edges. Second, the mock expectations in the MCP runtime tests needed multiple passes because the actual `client.callTool` signature shape differed slightly between the blocking and async paths. That should have been anticipated earlier. Third, the first pulse-refresh command hit the exact heredoc parsing problem the skill instructions warned about, which is mildly embarrassing because the safe form was already provided. I recovered quickly, but it shows that execution discipline matters just as much in maintenance notes as it does in product code. Overall I’m happy with the session, but I want to keep reducing avoidable rework in the validation phase.

## Lessons Learned
- If a repo already has one solid async lifecycle, prefer wrapping new APIs around it rather than creating a second persistence model.
- A subagent is not an async architecture; it is only useful when paired with a real start-and-poll protocol.
- Verification friction often shows where public API shapes were still mentally tied to the old implementation.

## Next Steps
- Decide whether to document the new async worker tools more prominently in user-facing examples.
- Consider whether the sync worker tools should warn callers more explicitly when used for long-running jobs.
- Revisit full MCP task augmentation only if host compatibility or cancellation/progress semantics become a hard requirement.
