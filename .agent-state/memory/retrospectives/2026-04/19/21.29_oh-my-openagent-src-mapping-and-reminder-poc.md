## AI Diary

I spent this session moving from intuition and repo claims toward things I could actually defend. That mattered more than it first looked. Early on, it would have been easy to keep treating the current project’s docs and code comments as if they already proved the callback/reminder story, but that would have been dishonest. The useful shift was accepting that the repo’s stated behavior is only a hypothesis until I can watch it happen in a real host/session path. That let me separate a successful in-process callback proof from the still-unproven host-integrated reminder path without collapsing the two into one fuzzy success story.

The second half of the session felt much stronger. I stopped trusting generated learn summaries and went back to `origin/src` for `oh-my-openagent`. That changed the quality of the analysis completely. Once I read the composition root, manager wiring, tool registry, continuation hooks, background-agent manager, tmux session manager, and atlas files directly, the architecture stopped feeling like a bag of hooks and started reading like a layered orchestration system. Writing the docs after that felt more like extraction than invention. Even with a negative live reminder result and a weekly session trend showing -45.5%, the work felt useful because the uncertainty got narrower. I now have a much more defensible picture of where orchestration really lives, where reminders are merely claimed, and which parts of the design we should adapt versus verify again.

## Honest Feedback

The biggest friction point was path reliability in my own terminal checks. I tripped over the same duplicated-path typo several times while reading tmux panes or source files, which wasted time and made the flow feel sloppier than it should have. That is not a deep systems problem, just an execution-discipline issue on my side, but it materially slowed the session and made some checks noisier than they needed to be.

The second friction point was the gap between “proof in process” and “proof in host.” The callback/reminder POC worked once I controlled the environment tightly with a temp result store and synthetic `promptAsync` capture, but the live OpenCode session still failed to show a reminder coming back into the main session. That is a good lesson, but it is also frustrating because it means I cannot honestly collapse the result into a clean success. The architecture path looks promising, but the live host path remains partially unverified.

The third friction point was that source-grounded analysis takes real time. It was tempting to rely on generated AGENTS-style summaries for `oh-my-openagent`, especially because they were conveniently structured, but doing so would have violated the rule we just established about trusting `origin/` first. Reading source directly is slower and heavier, but in this session it was the difference between a persuasive narrative and a defensible one. I am glad I paid that cost, but it does mean deep architecture work will continue to feel slower unless I keep producing better local docs as I go.

## Lessons Learned

- A callback/reminder path can be proven in-process without proving it live in the host; those are separate confidence levels and must not be blurred together.
- `oh-my-openagent` becomes legible once it is read as a layered system: composition root, runtime managers, tool surface, hook policy, host adapter.
- `BackgroundManager` is the execution orchestration core, `TmuxSessionManager` is the runtime substrate owner, and `Atlas` is the continuation/control-policy layer.
- Generated learn summaries are navigation aids only; source in `origin/` remains the only reliable basis for architecture claims.
