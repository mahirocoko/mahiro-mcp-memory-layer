## AI Diary
I spent this session correcting a deeper mistake in posture rather than just patching code. The earlier versions of the orchestration work looked convincing at the code-structure level, but they still behaved too much like worker invocation with polling instead of the spawned-subagent pattern the repo was supposed to copy from oh-my-openagent. That mismatch only became obvious once the runtime behavior was exercised from a fresh session and the human pushed back on what “done” really meant. I had to stop trusting the cleanliness of the architecture alone and start treating the live behavior as the real truth source.

The most valuable shift was moving orchestration entry onto the plugin-native path with session context, so reminders could return to the same main session instead of pretending MCP-only tools could emulate that loop by themselves. After that, the hard part became Gemini normal mode in tmux. It was easy to get fooled by false completions, trust prompts, prompt echoes, and sessions that looked idle while the pane still had state. I had to peel those apart one by one: remove `--screen-reader`, handle trust prompts, separate first-run startup from follow-up input, stop believing completion tokens for Gemini, and use pane behavior as the truth source. The final proof felt much more earned than the earlier “tests pass” confidence. The result is not just cleaner code; it is behavior that is far closer to the repo doctrine and to the human’s intent.

## Honest Feedback
First, I was too willing earlier in the broader session to treat internal structure and green tests as a substitute for runtime behavior. That was a mistake, and this session only worked well once I stopped trying to defend the earlier interpretation and instead accepted that the human’s behavioral definition of “done” was the correct one. Second, Gemini normal mode through tmux is tricky in ways that are easy to underestimate. Trust prompts, input timing, prompt echo, and output cadence all conspire to create false signals. A lot of the effort here came from undoing assumptions that looked reasonable but were not grounded in actual pane output. Third, the remaining repo surface still contains a lot of change at once, which makes confidence slower to rebuild. Even though the path now behaves much more honestly, it still requires discipline to keep the product contract narrow and to keep dead legacy patterns from creeping back in. This session improved that significantly, but the burden of proof remains high.

## Lessons Learned
- In this repo, “orchestration done” must be judged from runtime behavior in a fresh session, not just source topology.
- Gemini normal mode in tmux should be treated as a worker with its own execution semantics, not forced into Cursor-shaped assumptions.
- Plugin-native session context is essential when the architecture requires reminder return to the main session.
