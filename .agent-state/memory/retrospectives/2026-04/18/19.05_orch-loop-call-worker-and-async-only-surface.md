## Architecture Decisions

- Kept the plugin operator loop **thin** and session-scoped instead of introducing a second orchestration engine.
- Treated `session.promptAsync` as a valid session-visible continuation surface, making reminder delivery work even without a host-native reminder panel.
- Added `call_worker` as a lane-first explicit invoke tool rather than jumping straight to model-first public APIs.
- Removed public sync direct-worker MCP tools and made `orchestrate_workflow` start behavior async-only to reduce surface ambiguity.
- Preserved `wait_for_orchestration_result` as a read helper, not as a contradictory sync start path.

## AI Diary

I felt this session as a gradual tightening of edges rather than a single feature build. At first the orchestration story looked close to done because the pieces were already there: task starts, polling, reminders, runtime capability checks. But the deeper I looked, the clearer it became that we still had several “almost” states. `orch: on` almost behaved like a real orchestrator mode, reminders almost behaved like OMOA-style callbacks, direct invocation almost existed, and the async story was almost clean while sync MCP worker tools still lingered in the public surface. The work became about turning “almost” into “actually,” one seam at a time.

What I liked most was that the repo did not need a grand rewrite. The existing primitives were good enough. The operator ledger already gave us a place to represent control-plane truth. The orchestration result store already gave us execution truth. `session.promptAsync` ended up being the missing bridge that let the plugin behave more like a local orchestrator instead of a passive observer. Once that clicked, the rest of the session became a matter of aligning contracts: route user intent into actual async work, expose an explicit lane-first direct invoke, then remove contradictory sync behavior so the docs and runtime stop sending mixed signals.

The part that felt most instructive was the global-path acceptance testing. It forced a clearer distinction between repo-side readiness and host/runtime instability. I had to keep separating “our contract is correct” from “the host session still crashed before proving it.” That separation matters. It kept me from over-claiming success while still letting me finish the repo work cleanly.

## Honest Feedback

The biggest friction point was how easy it is to blur together plugin path, active project path, and runtime path when testing OpenCode interactively. Several times the evidence looked contradictory until I traced which session was actually loaded and which repo the app had resumed. That cost time and made the proof work noisier than it should have been. The second friction point was contract drift: each time a public surface changed, at least three different layers had to be updated together — registration, capability reporting, and docs/tests. The repo is disciplined enough for this to be manageable, but it still creates lots of small alignment chores. The third friction point was the host/runtime instability on global acceptance tests. The `Transforms cannot be re...` crash meant I could not cleanly finish the real-world validation story even when the repo-level implementation was solid. That is frustrating because it leaves a small confidence gap that is not really caused by the code under active development.

## Lessons Learned

- A thin operator loop can feel OMO-like without becoming a second orchestration engine if reminder continuation, resume, and verification all stay grounded in the same result store.
- `session.promptAsync` is good enough to count as a real session-visible continuation surface for plugin-local reminders.
- Public orchestration surfaces become easier to reason about once all starts are async-only.
- Lane-first direct invocation (`call_worker`) is a better first step than model-first public APIs because it keeps the contract stable while still giving humans explicit control.
- Interactive proof needs explicit project paths or the evidence chain becomes unreliable fast.
