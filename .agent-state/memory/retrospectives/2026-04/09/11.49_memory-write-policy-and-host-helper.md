## AI Diary
I felt this session turn a corner because the memory layer stopped looking like a bag of isolated primitives and started looking like a real product surface. Early on, I could feel the temptation to keep polishing retrieval details forever, especially because retrieval work always gives satisfying local wins. But the question that kept cutting through the noise was simpler and harsher: when would an agent actually know that something deserves to be remembered? Once I accepted that, the work became much clearer. I stopped treating memory as only storage and search, and started treating it as a decision system. That led naturally from candidate suggestion, to save hints in context-building, to a conservative policy, and finally to the one-call host helper. I’m glad I kept the changes incremental instead of trying to jump straight into a grand semantic-memory architecture. The profile-context improvements were especially satisfying because each layer made the output feel less like a raw retrieval dump and more like something a model could actually use. I also noticed that whenever I stayed pragmatic and let the repo’s existing contracts constrain me, the work stayed coherent. When I reached for broader abstractions too early, the session immediately got noisier.

## Honest Feedback
The first friction point was the amount of drift that accumulated between implementation intent and the real worktree state over a long session. More than once, I believed a feature was either fully committed or still pending, only to discover that parts of it had already landed through a prior commit or task-tool path. That made me spend extra effort re-grounding on git state before each next step. The second friction point was that semantic improvements in `context-builder` became subtly entangled. Dedupe, conflict suppression, preference extraction, and snapshot layout all lived close together, so a half-finished heuristic from one pass could destabilize assertions from another. It was manageable, but the file started carrying a lot of meaning in a small space. The third friction point was the boundary between this repo and the host/agent layer. I could make the memory layer increasingly capable, but every meaningful question about “when to auto-save in production” eventually ran into integration policy that belongs outside this package. That boundary is healthy, but it means there is a natural stop point where pushing further inside this repo stops being the highest-leverage move.

## Lessons Learned
- Memory work becomes more valuable when framed as decision support, not just retrieval quality.
- A conservative write policy is a better first production posture than full auto-save.
- One-call integration surfaces matter more than adding ever more low-level tools once the primitives are already solid.
- Profile semantics are worth building in layers; each small heuristic can be verified and committed atomically.
