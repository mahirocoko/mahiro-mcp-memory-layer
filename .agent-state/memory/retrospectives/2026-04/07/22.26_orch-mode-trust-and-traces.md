## AI Diary

I felt this session was less about writing code and more about slowly aligning behavior, doctrine, and implementation until they stopped contradicting each other. At several points I noticed that I was still carrying an older mental model: Gemini as mostly a summarizer, Opus as almost planning-only, delegate-first as a loose preference rather than a strict discipline. The user kept correcting those mismatches, and that was useful because it exposed where my behavior and the written rules were still drifting apart. I had to accept that some of the failure modes were not technical in the narrow sense, but procedural: picking the wrong model for planning, being too casual about what `orch:` should imply, and not making the trust problem explicit early enough. The traces were especially grounding. They showed that the system is already alive enough to produce real evidence, not just intentions. They also showed how easy it is to claim that orchestration is working while still missing subtler problems like misleading `completed` semantics or missing trace rows. I ended the session feeling more confident in the direction, but also more aware that orchestration quality lives or dies on discipline, not just on tools.

## Honest Feedback

The session made real progress, but it also surfaced three friction points clearly. First, the doctrine was initially too fuzzy, which let me drift into behaviors the user did not actually want. "Delegate-first" is not strong enough if I still instinctively step in as an implementer when the path feels obvious. Second, the trust issue is a serious operational paper cut. A single missing `trust: true` can make an orchestration run look flaky even when the real problem is a gating prompt, and that kind of failure can easily be misread if the trace consumer is not careful. Third, observability semantics still need work. A workflow-level `status: completed` that can coexist with `failedJobs: 1` is technically understandable, but it is human-hostile and encourages bad dashboards. I also noticed a softer friction point: our docs kept improving, but only because the user repeatedly corrected routing assumptions and sticky-mode semantics. That is a sign that the written system was not yet self-explanatory enough. The strongest lesson is that orchestration needs hard protocol wording, not polite suggestions.

## Lessons Learned

- Orchestrator quality depends more on explicit posture rules than on having more worker types.
- `orch:` must be treated as a behavioral contract, not just a stylistic hint.
- Trust is part of orchestration reliability, not an incidental CLI detail.
- Workflow-level terminal status must be interpreted alongside per-job counters, not in isolation.
- A single-file `AGENTS.md` can work, but only if it is internally coherent and concrete.

## Next Steps

- Tighten the interpretation of orchestration traces so success is measured with job counters, not only workflow status.
- Add a small trust protocol for Cursor-family workers so known trusted repos and unknown repos are handled differently.
- Keep testing `orch:` across unrelated repos to see whether the doctrine generalizes or still leaks local assumptions.
- Decide later whether sticky `orch: on/off/status` should become a real session-scoped implementation or remain procedural only.
