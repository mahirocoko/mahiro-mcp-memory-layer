## AI Diary
This session felt like the real closure that the earlier rounds only pointed toward. I had already improved the contract language, added blocking helpers, and even proven parts of the orchestration path live, but the user was right to keep pushing on the distinction between “better wording” and “the actual mechanism.” That was the uncomfortable part: I had to admit that the story was still incomplete. Once I stopped defending the half-solution, the architecture got cleaner. The requirement was not “make waiting nicer”; it was “make background sleep/check real.” That led directly to the right split: `workflow_*` remains the real job, `supervisor_*` becomes the detached watcher, and `get_orchestration_supervision_result` becomes the polling surface for supervision state rather than the raw workflow state.

Watching `opencode run` start a workflow, spawn a `supervisor_*` request, poll it, and reach `completed` made the feature feel settled in a way that unit tests alone never do. ## Honest Feedback
The biggest honest criticism is that I should have converged on this shape earlier. I spent too long refining intermediate solutions — first a blocking wait helper, then a blocking supervisor helper, then better docs around those helpers — before finally landing on the thing the user had been pointing at all along. That was not wasted work, but it did show a tendency to optimize the current abstraction instead of stepping back and asking whether the abstraction itself was wrong. The second friction point was test design. Detached behavior is easy to implement one way and test another way, and I had to correct myself more than once when my tests overfit polling counts instead of asserting the real contract. The third friction point was the mismatch between “background relative to the MCP request” and “durable beyond process lifetime.” I had to be careful not to oversell what this repo now does. It solves the request-timeout problem the user asked about, but it is still in-process supervision, not a separate durable job queue. I’m happy with the result, but I would have preferred to get there with one fewer intermediate stage.

## Lessons Learned
- If a user keeps distinguishing between a nicer helper and the actual mechanism, assume the mechanism is still missing.
- Detached behavior needs a detached state model; trying to overload the original workflow result shape makes the contract fuzzy.
- For polling systems, tests should assert terminal states and stored records first, not brittle internal call counts.
- Oracle review is especially valuable near the end of orchestration work because the remaining bugs tend to be boundary holes, not core algorithm mistakes.
