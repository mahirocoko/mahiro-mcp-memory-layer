# Session Retrospective

## Session Summary
วันนี้ผมปิดงานบั๊ก memory plugin ที่ไม่โตเมื่อ runtime ส่ง `message.part.updated` เป็นหลัก แทนที่จะส่ง `message.updated` ตามที่ adapter เดิมคาดไว้ ระหว่างเริ่มเขียน retrospective มี pulse context แบบ pre-write อยู่ที่ 12 retrospectives สะสม, วันนี้ยังนับเป็น 0 session, streak 0 วัน, branch ปัจจุบันคือ `main` และมีไฟล์เปลี่ยนค้างอยู่ 19 รายการ ซึ่งยังไม่รวม retrospective ฉบับนี้เข้าไปในตัวเลขนั้น

## Timeline
- ตรวจ root cause ของ memory ไม่โตจนยืนยันได้ว่า core memory ยังทำงาน แต่ OpenCode plugin adapter ผูกกับ `message.updated` และ `session.idle` แน่นเกินไป
- เปรียบเทียบแนวทางกับ `oh-my-openagent` และสรุปว่า Claude Code hooks เหมาะเป็น reference เรื่อง event/session plumbing มากกว่าเป็นคำตอบตรงของ semantic memory
- วาง acceptance criteria ชัดเจน: `message.part.updated` ต้องอัปเดต turn state และ precompute ได้ แต่ห้ามขยาย write boundary ออกนอก `session.idle`
- แก้ `runtime-state.ts` ให้ `message.part.updated` นับเป็น turn update, เพิ่ม extraction จาก `properties.part.text`, และรักษา stale-text semantics เดิม
- แก้ `runtime-shell.ts` ให้ generic `event` path รองรับ `message.part.updated` ด้วย wake-up + debounced precompute โดยไม่แตะ persistence flow
- เพิ่ม test ครอบ 3 เคสหลัก: debounce/cache, stale completion, และ textless part turn ที่ห้าม reuse ข้อความเก่า
- รัน diagnostics, targeted tests, full test suite, typecheck, และ build จบครบ

## AI Diary
รอบนี้ผมรู้สึกว่าปัญหามันหลอกตาเก่งมาก เพราะอาการที่เห็นคือ “memory ไม่โต” ซึ่งฟังเผินๆ เหมือน persistence layer หรือ policy save พัง แต่พอไล่ทีละชั้นจริงๆ มันกลับเป็นปัญหา adapter เชิง event contract มากกว่า ผมค่อนข้างพอใจที่ไม่ได้รีบกระโดดไปแก้ผิดจุด เช่น ไปยุ่งกับ conservative save policy หรือเปลี่ยน storage root ทั้งที่สาเหตุจริงอยู่ในทางผ่านของ event ก่อนถึง memory layer ด้วยซ้ำ

สิ่งที่ผมชอบใน session นี้คือการคุม scope ได้ค่อนข้างดี เราไม่ขยายงานไปถึงการออกแบบ host abstraction ใหม่ ไม่ย้ายไปพึ่ง Claude hooks และไม่เผลอทำ “fix ใหญ่เกินจำเป็น” แค่ทำให้ `message.part.updated` เข้า semantics เดียวกับ `message.updated` ในชั้น state/precompute แล้วเก็บ `session.idle` ไว้เป็น write boundary เดิม ซึ่งเป็นการแก้ที่สั้นแต่ตรงจุด ผมยังชอบที่ evidence ครบมาก: มี direct code reading, มี external comparison กับ `oh-my-openagent`, มี Oracle ช่วยยืนยันทิศ, แล้วสุดท้ายมี test เป็นหลักฐานว่าพฤติกรรมใหม่ไม่หลุดไปทำ persistence ก่อนเวลา

แต่ผมก็รู้สึกชัดว่าระบบแบบนี้เปราะกับ contract mismatch มาก ถ้า host เปลี่ยน shape ของ event อีกนิดเดียวแล้ว adapter ไม่ normalize ให้ดี อาการภายนอกจะดูเหมือน “memory เสีย” หรือ “ระบบไม่จำ” ทั้งที่ความจริงคือ data ไม่เคยถูกยกขึ้นมาเป็น canonical turn ตั้งแต่แรก ตรงนี้เตือนผมว่าของพวกนี้ต้องแยกให้ออกระหว่าง event ingestion, turn normalization, และ semantic memory write ถ้ายังปล่อยให้สามชั้นนี้ไหลปนกัน เวลาเสียจะเสียแบบชวนเข้าใจผิดมาก ตอนทำ retrospective นี้แม้ weekly session change จะดูเป็น +1000% จากฐานย้อนหลังที่เล็กมาก แต่ผมไม่อยากหลงกับเลขสวยๆ นั้น เพราะสิ่งสำคัญกว่าคือเราได้ปิดบั๊กด้วยโมเดลความคิดที่แม่นขึ้น ไม่ใช่แค่ให้ test ผ่าน

## Honest Feedback
ผมยังมีจุดหงุดหงิดกับ session นี้อยู่สามเรื่องชัดๆ เรื่องแรกคือ shell ergonomics ตอนทำ `/rrr` เองยังมีสะดุดกับ heredoc parsing ใน bash call เดียว ทั้งที่ skill ก็เตือนแล้วว่ารูปแบบนั้นเปราะ แปลว่าผมยังประเมิน friction ของ host shell ต่ำไปนิดและต้องเปลี่ยนไปใช้ `python3 -c` แบบปลอดภัยกว่าในรอบแก้ไข เรื่องที่สองคือ git working tree ตอนนี้มีงานค้างจากหลายก่อนหน้าอยู่เยอะ ทำให้ retrospective ต้องระวังมากว่าจะไม่เหมารวม diff ทั้ง branch เป็นผลงานของ session นี้ ถ้าไม่มีวินัยตรงนี้ retrospective จะเริ่มโกหกแบบไม่ตั้งใจ เรื่องที่สามคือแม้เราจะแก้ root cause ปัจจุบันได้แล้ว แต่ plugin event model ยังพึ่งพาความรู้เฉพาะว่า event ไหนนับเป็น turn update ซึ่งถ้า host มี event ชนิดใหม่อีก เราอาจกลับมาเจออาการลักษณะเดิมอีกรอบได้

ผมยังรู้สึกว่าการแยก “research mode” กับ “implementation mode” ทำได้ดีขึ้น แต่ยังมีความล้าเล็กๆ จากการต้องสลับบริบทหลายรอบ: memory bug, architecture tradeoff, `/project`, `/learn`, กลับมาลง patch, แล้วมาปิดด้วย `/rrr` ถ้าผมไม่คุมเส้นเรื่องให้ชัด session นี้อาจกลายเป็นหลายงานปนกันจนโฟกัสเสีย โชคดีที่สุดท้ายยังรักษาแกนกลางไว้ได้ว่าเป้าหมายคือทำให้ OpenCode-native memory path กลับมาทำงานตามจริงโดยไม่เปลี่ยน product direction

## Lessons Learned
- อาการ “memory ไม่โต” ไม่ได้แปลว่า memory core พังเสมอไป บ่อยครั้งเป็นปัญหาที่ event ingestion layer ก่อนถึง persistence
- ถ้า host ส่ง event แบบ incremental (`message.part.updated`) ต้อง normalize ให้เป็น canonical turn update ก่อนค่อยคุยเรื่อง memory
- ควรเก็บ `session.idle` เป็น write boundary ต่อไป เพราะช่วยกัน partial-save ได้ดีและทำให้ reasoning ของระบบง่าย
- การเทียบกับ repo อื่นมีประโยชน์มาก แต่ต้องแยกให้ออกว่าอีก repo เก่งเรื่อง transcript/session plumbing ไม่ได้แปลว่าเหมาะจะเป็น semantic memory architecture

## Next Steps
- พิจารณาว่าควรเพิ่ม abstraction เล็กๆ สำหรับ “turn text update event” เพื่อกัน future event-shape drift หรือไม่
- ถ้าผู้ใช้ต้องการ ให้สร้าง commit สำหรับ patch นี้แบบ atomic ตามที่ verify แล้ว
- หากยังพบอาการ memory ไม่โตใน runtime จริง ให้เก็บ live event trace ต่อเพื่อดูว่ามี completion boundary แปลกกว่านี้อีกหรือไม่
