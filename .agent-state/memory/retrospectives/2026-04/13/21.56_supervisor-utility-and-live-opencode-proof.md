## Honest Feedback
I still had to correct myself midstream more than I would like. The biggest miss was that I initially treated the background-first contract change as “done” before adding the actual supervisor utility. In hindsight, that was the same category of mistake as earlier in the day: improving the narrative before fully closing the mechanism. Another friction point was that I needed Oracle to spot one avoidable API leak on the first implementation of the supervisor: reusing the wait schema exposed `includeCompletionSummary` even though the supervisor always returns a summary. That was not a severe bug, but it was exactly the sort of contract noise that makes an API feel less intentional. The third friction point was operational: driving OpenCode through tmux required a bit of shell-level discovery first, because the panes were just sitting at shell prompts and not at a clearly labeled interactive state. It worked, but the setup was more manual than ideal. Overall, I’m happy with the result, but this session reinforced that “repo-owned utility” should mean truly end-to-end: mechanism, docs, tests, and live proof.

## Lessons Learned
- A background-first contract is incomplete until the repo owns an actual supervisor path, not just a recommendation.
- Shared schemas are useful, but they can leak dead flags across neighboring tools unless the boundaries are checked carefully.
- For orchestration work, live OpenCode smoke tests add a different kind of confidence than unit tests because they validate host/tool wiring, not just local logic.
- The result store remains the right source of truth for both raw polling and higher-level supervision; adding parallel state would have made the design worse.
